# Amazon Bedrock Provider via the Converse API (Model-Agnostic, SigV4)

## Decision

Gini ships a `bedrock` provider distinct from `anthropic`. It speaks AWS's **Converse API** (`POST https://bedrock-runtime.{region}.amazonaws.com/model/{modelId}/converse[-stream]`), the model-agnostic message API that works across every Bedrock model family — Claude, Amazon Nova, Meta Llama, Mistral, DeepSeek, and more — with one request/response shape and unified tool use. Auth is **AWS SigV4** (service `bedrock`) computed from a long-term IAM access key the user **enters when adding the provider**: an `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` pair. Gini writes them to `~/.gini/secrets.env` (mode 0600) under those standard names and reads them from the environment at call time. Gini does **not** read `~/.aws`, SSO session caches, `~/.aws/config` role chains, or instance roles — credential resolution is env-only (`resolveAwsCredentials`).

The `modelId` is a Bedrock model id sent verbatim in the request path — typically a cross-region inference-profile id (e.g. `us.amazon.nova-pro-v1:0`, `us.anthropic.claude-opus-4-8`, `us.meta.llama3-3-70b-instruct-v1:0`). There is no Anthropic-style id remapping.

`bedrock`'s catalog `auth` is `"aws"`. Unlike the API-key providers, gini holds no single bearer for it — it stores the access-key pair and SigV4-signs each request. The web Add/Edit forms show AWS Access Key ID + Secret Access Key fields (required on add, optional on a model/region-only edit), and `gini setup` prompts for the pair. Because gini stores those keys, the Settings row is removable (disconnect scrubs `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` from `secrets.env` and `process.env`, like the API-key providers); credential rotation is "re-enter the keys" through the Edit dialog.

## Context

Claude-in-Bedrock has an Anthropic-Messages-compatible endpoint, but it only serves Claude. To use Nova, Llama, Mistral, DeepSeek, etc., the right surface is the Bedrock Runtime **Converse** API, which AWS designed as "write once, run against any Bedrock model." So the model-agnostic requirement drives a Converse transport rather than reusing the Anthropic Messages path. SigV4 over a long-term IAM access key (vs. a Bedrock API key) needs no key minting and requires no `iam:CreateServiceSpecificCredential` permission. Gini takes the keys by explicit entry rather than auto-reading `~/.aws`: a from-nothing user has no AWS CLI or credentials file to read, and entering the pair keeps the trust boundary identical to the API-key providers (gini owns what's in `~/.gini/secrets.env`, nothing else on disk).

## Wire Shape

`callBedrockConverse` mirrors `callAnthropicMessages`: it streams when an `onDelta` callback is present and parses the JSON body otherwise. Translation maps Gini's OpenAI-shaped transcript onto Converse:

- **System hoist** — `system` messages concatenate into the top-level `system: [{text}]` array.
- **Tool use** — assistant `tool_calls` become `{toolUse:{toolUseId,name,input}}` content blocks; tool defs map to `toolConfig.tools[].toolSpec = {name, description, inputSchema:{json}}` with `toolChoice:{auto:{}}`.
- **Tool results** — consecutive `tool` messages group into one user message of `{toolResult:{toolUseId, content:[{text}]}}` blocks.
- **Tool-less turns over tool history** — Converse hard-rejects `toolUse`/`toolResult` blocks unless the request also carries a `toolConfig`, and `toolConfig` itself rejects an empty `tools` array. So whenever no `toolConfig` is sent — a tool-less turn (the iteration-cap summary call advertises no tools to force a text answer) or a model that rejects tool use (e.g. DeepSeek) — `translateMessagesToConverse` flattens the history's tool blocks to plain text (`Called tool name(args)` / `Tool result: …`) rather than emitting structured blocks. This keeps the model grounded in what it did without re-opening the tool channel and without the `toolConfig`-required 400. Invariant: a Converse request carries tool blocks **iff** it carries a `toolConfig`.
- **Multimodal** — `image_url` data URLs become `{image:{format, source:{bytes}}}` (the base64 string is the blob in JSON); `format` is a bare token (png/jpeg/gif/webp). Per-model vision/doc support is gated in `resolveProviderModality` (Claude 3+, Nova Pro/Lite/Premier, Pixtral, Llama 4, Llama 3.2 vision; text-only ids stay false).
- **max_tokens** — sent as `inferenceConfig.maxTokens`. The default depends on the send mode: a **streaming** turn resolves the model's real output ceiling via `resolveMaxOutputTokens` (e.g. 128K for Claude Sonnet/Opus 4.6+) so a large tool argument isn't truncated mid-JSON, while a **non-streaming** turn keeps the conservative `DEFAULT_ANTHROPIC_MAX_TOKENS` floor. A user-pinned `extraBody.max_tokens` and the vision per-call override both take precedence. No `extraBody` key is forwarded to Converse; the only `additionalModelRequestFields` Gini sets is the fine-grained-tool-streaming beta below (general `extraBody`→`additionalModelRequestFields` passthrough remains out of scope).
- **Non-stream** — `parseConverseResponse` reads `output.message.content[]` (text + `toolUse`), maps `stopReason`, and normalizes `usage.{inputTokens,outputTokens,totalTokens}` into the snake_case keys `estimateCost` reads.
- **Streaming** — `converse-stream` returns `application/vnd.amazon.eventstream` (binary framing: `[4B totalLen][4B headersLen][4B preludeCRC][headers][payload][4B msgCRC]`). `readConverseStream` parses frames by length (CRCs unvalidated), reads the `:event-type` / `:message-type` string headers, and handles `messageStart`/`contentBlockStart`/`contentBlockDelta`(text + `toolUse.input` fragments)/`contentBlockStop`/`messageStop`(stopReason)/`metadata`(usage), surfacing any exception frame as a thrown error.
- **Fine-grained tool streaming** — on a streaming turn that carries a `toolConfig` against a Claude 4-family model (`claudeSupportsFineGrainedToolStreaming`), the body sets `additionalModelRequestFields.anthropic_beta = ["fine-grained-tool-streaming-2025-05-14"]`. Without it Bedrock buffers a `toolUse` block's entire input JSON server-side and emits nothing on the wire until the whole argument is generated; a large argument (a long file body written inline) then leaves the Converse event stream idle past the runtime's socket timeout, surfacing as `TimeoutError: The operation timed out.` (the idle/stall timeout + transient retry below now recover this residual stall instead of failing the turn outright). The flag streams the tool input incrementally (one key/value at a time), keeping the connection fed. It is gated to the Claude 4 family — Claude 3.x and the non-Claude families (Nova, Llama, DeepSeek, Mistral) don't accept it and could 400 — and streaming-only (the non-stream `/converse` path holds the whole response anyway). NOTE: this is the Bedrock mechanism; the first-party Messages API carries the same beta as the HTTP `anthropic-beta` header instead (see ADR anthropic-messages-provider.md), and Converse silently ignores the tool-level `eager_input_streaming` property the first-party API also accepts. (AWS warns the streamed input may be partial/invalid JSON if `stopReason` is `max_tokens`; the loop's existing "invalid JSON arguments" handling absorbs that, and the resolved max-output ceiling makes truncation unlikely for normal writes.)
- **Streaming `max_tokens` clamp** — the raised streaming ceiling (`resolveMaxOutputTokens`) is clamped DOWN at send time so `estimated_input + max_tokens` fits the model context window whenever there is room (`clampStreamingMaxTokens` in provider.ts, shared by both the Converse and first-party send paths). It only ever lowers the value (never raises it) and never shrinks prior-context replay — it reduces the OUTPUT budget. On an already-near-full prompt it floors at a small positive output budget (1024) rather than going to zero/negative, so in that edge case the sum can still exceed the window — that case is the existing context-overflow compaction path's job, not the clamp's. For an ordinary prompt this prevents a 400 on `input + max_tokens > window` before a token streams.
- **Idle/stall stream timeout + transient retry** — the streaming readers (`readConverseStream`, `readAnthropicMessagesStream`) wrap each `reader.read()` in `readWithIdleTimeout`: if no bytes arrive for `IDLE_STREAM_TIMEOUT_MS` the reader is cancelled (tearing down the socket) and a distinguishable `StreamIdleTimeoutError` is thrown — NOT an `AbortError` (so it is never confused with a user cancel) and NOT a context overflow. The chat-task model-call loop classifies that error (and OS/socket faults like "operation timed out"/"fetch failed"/"connection reset") as transient via `isTransientModelError` and retries with bounded exponential backoff (`MAX_TRANSIENT_RETRIES`), resetting the per-attempt streamed/route state exactly as the overflow path does, before falling through to the existing hard-fail on exhaustion. A turn cancel during backoff still bails immediately. This is defense-in-depth: the fine-grained-streaming flag removes the specific Bedrock buffering stall, and the idle-timeout-plus-retry recovers any other mid-stream stall (provider hiccup, network blip) instead of hanging until the OS socket timeout fails the task.

Dispatch arms route `bedrock` in `generateToolCallingResponse`, `generateTaskSummary`, `generateStructured` (via `callBedrockStructured`, prompt-for-JSON), and `generateVisionAnalysis`.

## Auth and Trust Boundary

`config.json` stores only `awsRegion` and the (informational) `baseUrl` — never any credentials. The access key + secret are written to `~/.gini/secrets.env` on add (the same store the API-key providers use) and read back from `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` in the environment at call time; `resolveAwsCredentials` reads only those env vars (`AWS_SESSION_TOKEN` is still honored if present in the env, for a user who exports a temporary session that way, but the provider's entry path is long-term keys). `signAwsRequest` (`src/aws-sigv4.ts`, `node:crypto`, no AWS SDK) computes the canonical request → `kDate→kRegion→kService→aws4_request` signing key → signature, emitting `Authorization`/`x-amz-date`/`x-amz-content-sha256` (and `x-amz-security-token` when a session token is present). The canonical URI **double-URI-encodes** the path (non-S3 SigV4 rule), so a model id's `:` (`…v1:0` → wire `%3A` → canonical `%253A`) signs correctly. `redactSecrets` masks `AKIA…`/`ASIA…` access-key ids and the computed `Signature=…` hex so neither leaks through `providerHealth` / `/api/status` / trace records. Region resolves from explicit `awsRegion`, else `AWS_REGION`/`AWS_DEFAULT_REGION`, else `us-east-1`.

`setSetupProvider`'s bedrock branch requires the access key + secret on a fresh add (optional on a same-provider edit whose keys already resolve, for a model/region-only save), rejects a half-entered pair, writes the pair to `secrets.env` + `process.env`, and persists model + region. The web Add Provider form's bedrock panel shows the two key fields, a free-text model id, and an optional region; `gini setup` prompts for the pair; `gini provider set bedrock [model] [--aws-region]` sets only model + region (keys are entered through the form or `gini setup`).

## Test Surface

- `src/aws-sigv4.ts` / `src/aws-sigv4.test.ts` — the signer in isolation: golden vector, temporary-credential `x-amz-security-token`, default-`now`, env-only credential resolution (returns null when the env keys are absent — no `~/.aws` fallback), and reserved-character canonical-URI encoding.
- `src/provider.test.ts` — fetch-mocked bedrock tests: SigV4-signs the Converse URL (`%3A`-encoded model id, no `x-api-key`) with the converse body shape, system hoist + `toolConfig` + `toolUse` parse, the tool-less summary turn flattening history tool blocks to text (no `toolConfig` 400), the `converse-stream` event-stream parser (text deltas, tool-use, stopReason, usage) via an event-stream encoder, an exception frame, a non-OK error, `providerHealth` configured/not + no-credentials error, the vision (image) path with region default, structured-output JSON, and fine-grained tool streaming gating (the `anthropic_beta` flag present on a streaming Claude tool turn; absent on a tool-less turn, a non-streaming/structured turn, and a non-Claude model).
- `src/provider-capabilities.test.ts` — `claudeSupportsFineGrainedToolStreaming` accepts Claude 4-family ids (bedrock-prefixed or bare first-party) and rejects Claude 3.x / Nova / Llama / DeepSeek / Mistral / unknown.
- `src/runtime/setup-api.test.ts` — bedrock writes the entered key pair to secrets.env + `process.env` (persisting model + region), accepts a keyless edit when keys already resolve, and rejects a fresh add with no/half-entered keys.
- `src/cli/setup.test.ts` — `gini setup` bedrock prompts for and saves the access key + secret when the env has none, captures region without re-prompting when keys are present, and aborts when a key field is left blank.

## Out Of Scope, Linked Follow-Ups

- Full IAM-role / `AssumeRole` sourcing, IMDS/container credentials, `~/.aws/config` SSO, and auto-reading the `~/.aws/credentials` profile. The signer uses the long-term access key the user entered (stored in `~/.gini/secrets.env`, read from the `AWS_*` env). A user on a temporary session can still export `AWS_*` (incl. `AWS_SESSION_TOKEN`) into the launching shell, but there is no `~/.aws` auto-detection.
- A live model picker (`ListFoundationModels` / `ListInferenceProfiles`); the catalog ships a cross-family starter list and the UI accepts any id.
- Bedrock Guardrails, prompt-management resources, and general user-controlled `additionalModelRequestFields` passthrough (the runtime sets only the fine-grained-tool-streaming beta described under Wire Shape).

## Consequences

- Any Bedrock model — not just Claude — is reachable end-to-end (tool use, streaming, vision) signed with the long-term AWS access key the user entered, no API key and no key minting. Verified live against `us.amazon.nova-lite-v1:0` (a non-Anthropic model) through a real chat turn.
- `src/aws-sigv4.ts` is a dependency-free, provider-agnostic SigV4 signer reusable by any future AWS-signed transport.
- `bedrock` and `anthropic` are independent providers; `anthropic` stays first-party-key-only (see ADR anthropic-messages-provider.md).
- A streaming Claude tool turn that emits a large inline tool argument (e.g. writing a long file body in one `file_write` call) streams that argument incrementally rather than stalling on Bedrock's server-side buffer, so it no longer trips the runtime socket timeout that the un-flagged path hits when the argument takes longer to generate than the idle limit. Verified live through a real chat turn writing a large file.
