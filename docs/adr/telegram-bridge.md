# Telegram Messaging Bridge

## Decision

A Telegram bridge is a `MessagingBridgeRecord` with `kind: "telegram"` and a per-bridge encrypted bot token. The runtime talks to `api.telegram.org` directly over `fetch` — no SDK dependency. Outbound (`POST /api/messaging/:id/send`) calls Telegram's `sendMessage`; inbound is a long-poll `getUpdates` loop owned by the gateway. Bot tokens live in the per-instance secret store under the `messaging.<bridgeId>` namespace and never appear on the bridge record.

The macOS Keychain, public webhooks, and a Telegram SDK are all rejected. They each add either a remote-headless dialog (Keychain), a public-internet surface (webhooks), or a runtime dependency tree (SDK) that the local-first Gini shape does not need.

## Context

Gini's messaging surface already modeled `MessagingBridgeRecord`, `MessagingMessageRecord`, and HTTP endpoints (`POST /api/messaging`, `/:id/send`, `/:id/receive`, `/:id/health`, `/:id/disable`). Until this ADR the telegram code path was a stub: `checkMessagingBridge` only verified that `TELEGRAM_BOT_TOKEN` was set in the runtime env, and `sendMessagingOutput` wrote a state record without performing a Bot API call. A real integration needed three things: secret-bound transport, an outbound call site, and an inbound pump.

Webhook delivery requires Telegram to reach the gateway over the public internet. The gateway binds to `127.0.0.1` per ADR `local-runtime-architecture.md`, so webhook mode would force the user to operate a tunnel (ngrok, cloudflared) and pin a public certificate. Long-poll `getUpdates` works from any network position and keeps the gateway entirely on the loopback interface.

A Telegram SDK (e.g. `node-telegram-bot-api`) is rejected on the same grounds the rest of this codebase prefers — the wire format is a small set of POST endpoints with JSON bodies; introducing a runtime dependency only adds upgrade exposure, a handler abstraction the runtime does not need, and (in the python-side reference systems we surveyed) elaborate workarounds for the SDK's own coupling to a process-wide `Application` lifecycle.

The Telegram Bot API caps the long-poll timeout at 50 seconds and uses an offset-based acknowledgment scheme: the next call's `offset` parameter implicitly acknowledges all updates with `update_id < offset`. The runtime therefore has to persist the next offset so a crash mid-batch doesn't replay messages that already produced tasks.

## Required Now

- `MessagingBridgeRecord` carries optional `secretRefs` (per-bridge encrypted secret pointers) and `metadata` (kind-specific free-form state — for Telegram: `botUsername`, `botId`, `lastOffset`).
- `POST /api/messaging` with `kind: "telegram"` requires a `botToken` field. The token is validated as header-safe printable ASCII (no whitespace, no control characters) before storage; without that gate a control-char token would be rejected at fetch time and the underlying error string would echo the full URL — including `/bot<token>/` — into the persisted `bridge.message`. The token is then written through `writeSecret(instance, "messaging.<bridgeId>", "bot-token", value)` and immediately discarded from memory.
- `POST /api/messaging/:id/health` performs a real `getMe()` round-trip when the bridge is Telegram-kind. Bot username/id land on `bridge.metadata` (merged into live metadata, not overwritten from a pre-await snapshot, so concurrent poller writes to `lastOffset` survive). Failures mark `bridge.status = "error"` with the API description as `bridge.message`. Any `/bot<token>/` residue is scrubbed before persisting via `sanitizeBridgeStatusMessage`. A user-initiated disable racing with the health check is preserved — the final mutate refuses to overwrite `status === "disabled"`.
- `POST /api/messaging/:id/send` dispatches `sendMessage` for Telegram-kind bridges. The `target` field is the `chat_id`. Send failures mark the resulting `MessagingMessageRecord.status = "failed"` with `error` set to the API description.
- Outbound text is rendered as Telegram MarkdownV2 by default. `src/integrations/telegram-format.ts` converts the common Markdown subset agents emit — fenced code blocks, inline code, and `**bold**` — into MarkdownV2 and escapes every other special character so the API cannot reject a payload over a stray `.` or `-`. Callers pass `parseMode: "none"` on the send body to opt out and send a literal string.
- The formatter hides code spans behind sentinel placeholders **before** matching `**bold**`, so bold runs that wrap an inline code span (a common agent pattern: ``**`git rebase`**``) are recognized as a single match instead of two orphan `**` markers. When a bold run wraps a single code token (with optional whitespace), the bold markers are dropped and only the code span renders — Telegram MarkdownV2 does not reliably support inline code nested inside bold, so dropping the bold preserves the agent's intent (highlight a command name) without provoking a malformed-entity error from the API.
- `POST /api/messaging/:id/send` accepts an optional `photo: { url | fileId | path }` field. The bridge dispatches `sendPhoto` instead of `sendMessage`; the caller's `text` becomes the caption, the MarkdownV2 transform applies to the caption, and the resulting `MessagingMessageRecord.media` carries the source descriptor. Local-path uploads use multipart/form-data so the bot can attach agent-produced files without staging them behind a public URL.
- `POST /api/messaging/:id/disable` deletes every encrypted file under `messaging.<bridgeId>` and clears `bridge.secretRefs`. The bridge status flips to `"disabled"` before the file cleanup so a partial failure cannot leave an active token on disk under a configured bridge.
- The gateway runs a Telegram poller supervisor that reconciles per-bridge long-poll loops against state every `GINI_MESSAGING_RECONCILE_MS` milliseconds (5s default; the legacy `GINI_TELEGRAM_RECONCILE_MS` env var is honored for backwards compatibility, as the same cadence drives the Discord supervisor). Loops run `getUpdates(offset, 25, signal)` with `allowed_updates: ["message"]`, route each message through `receiveMessagingInput`, and persist `metadata.lastOffset = update.update_id + 1` after each update. A missing or unreadable secret file flips the bridge to `status: "error"` (via the shared `markBridgeError` helper) so the supervisor drops it on the next reconcile — without that guard the loop would hot-restart on every tick because `shouldRun` only checks the secret-ref entry, not the file's presence on disk. The mark-error path refuses to overwrite an explicit `disabled` status.
- Each Telegram `chat_id` is bound to a persistent `ChatSessionRecord` tagged with `source: { kind: "telegram", bridgeId, chatId, target }`. `receiveMessagingInput` looks the session up (or creates it on first contact) and submits the user turn via `submitChatMessage` — the same path the web chat uses — so the agent sees prior turns and the chat-task loop runs with full session history. Demo / generic bridges keep the standalone-task path for CLI parity. The same key (negative `chat_id` for groups, positive for private DMs) handles both private chats and group/supergroup chats — one session per Telegram chat, shared across every member of the group.
- Group/supergroup chats add two transforms on the inbound path so multi-user threads stay readable: (a) the bot's own `@botusername` mention is stripped from the message text, including the `/cmd@botusername` form that Telegram clients auto-append when commands are issued in groups; (b) the sender's `@username` (falling back to `first_name`) is prefixed onto the task input as `<sender>: <message>` so the agent can address the right person and distinguish concurrent speakers. Other users' mentions in the same message stay intact. Outbound replies pass `reply_to_message_id` so the bot's response threads onto the question it's answering — without this, group conversations become unreadable as soon as more than one speaker is involved. The poller reads the bot's username from `bridge.metadata.botUsername` (populated by the first `getMe` health probe), so `gini messaging health <id>` is now a recommended first step before adding the bot to a group.
- A chat allowlist on `bridge.metadata.allowedChatIds` gates the inbound path. Strangers' updates are dropped silently — we don't acknowledge the bot's existence to chats the operator hasn't enrolled — and the offset still advances so denied updates can't accumulate on the Telegram side. **There is no trust-on-first-use**: every chat, including the operator's own first DM, is denied unless it claims the bridge's pairing code (see below) or has been explicitly enrolled via `gini messaging allow <bridge> <chat-id>`. TOFU was rejected because the bot's `@username` is public the moment it's created with @BotFather — a stranger watching for new bot accounts could otherwise claim ownership before the legitimate operator's first DM.
- A one-shot **pairing code** sits on `bridge.metadata.pairingCode` with a 15-minute expiry. Telegram doesn't expose `chat_id` in its UI, so the alternative (operator looks up their own id) is friction-prone; the pairing code resolves that by binding the first private DM that sends the code to the bridge. The poller's deny path tries `tryClaimPairingCode` before recording the denied attempt — on success it enrolls the originating chat, sets `ownerChatId` if unset, consumes the code, and sends a single "Paired." confirmation back. The pairing message itself is consumed (not turned into a task input). A wrong code from a private DM does not create a task or enrollment, but the bridge may still nudge the chat with the hint reply described in the next bullet so the operator isn't met with silence inside their own pairing window. Codes are 32-bit random hex with a `pair-` prefix; outside an active pairing window, denied attempts get no acknowledgment at all, so an attacker who finds the `@username` cannot tell whether the bot is alive. Group chats can never pair via code — only direct `allow` enrolls them, so a stranger adding the bot to their group cannot trip the shortcut. `gini messaging pair <bridge>` and `POST /api/messaging/:id/pair` rotate the code when the operator needs a fresh window.
- During an active pairing window, an unpaired private DM that does NOT carry a valid code gets a single hint reply ("Please send your pairing code to enroll this chat.") instead of complete silence. Without this the operator's first instinctive "hi" would land on a black hole and look like the bot is broken. The hint only fires inside a pairing window — outside it (no code minted, or after the code is consumed or expires) the bridge goes back to silent-drop so strangers who find the `@username` get no confirmation the bot exists. An explicit `allowChat` closes the pairing window for the same reason: once the operator has established trust through the CLI, the public-facing hint becomes noise.
- To make chat-id discovery painless (operators don't usually know their own Telegram user_id by heart), every denied attempt is recorded on `bridge.metadata.recentDeniedChats` — a bounded ring buffer of `{ chatId, chatType, sender, lastAttemptAt }` entries, deduped by `chatId` so a flood of pings from the same source can't bloat state. The operator DMs the bot once, runs `gini messaging chats <bridge>`, sees their own attempt in the pending list, and runs `gini messaging allow <bridge> <chat-id>`. Once enrolled, the entry is removed from the denied list so the view stays clean.
- `bridge.metadata.ownerChatId` records the first `chat_id` ever enrolled and stays stable across subsequent `allow`/`deny` calls. Removing the owner chat from the allowlist parks the bot but leaves `ownerChatId` intact as a historical record; future tooling (`gini messaging set-owner`) can rotate it explicitly.
- Allowlist administration surface: `gini messaging allow|deny|chats <bridge> [<chat-id>]` and `gini messaging pair <bridge>` over the CLI, and `POST /api/messaging/:id/{allow,deny,pair,reject-pending}` + `GET /api/messaging/:id/chats` over HTTP. `reject-pending` is web-only — it clears a row from `recentDeniedChats` without granting allowlist access, backing the operator UI's "Reject" button on a pending pairing request; the chat remains denied and its next message re-appears in the pending list. Both surfaces emit audit rows on every decision: `messaging.chat.allowed`, `messaging.chat.denied`, `messaging.pairing.minted` (operator rotated the code), `messaging.pairing.claimed` (a private chat consumed a valid code), and `messaging.pending.rejected` (operator dismissed a pending row without enrolling). The activity trail captures every enrollment outcome with `chatId` and (for pairing events) the expiry timestamp on `evidence`.
- When the chat-task settles, the poller's per-message worker calls `syncChatTaskResult` to land the assistant message in the chat session and dispatches the assistant text back via `sendMessagingOutput`. The MarkdownV2 transform runs over the outbound text on the way through. `[SILENT]` task summaries suppress the mirror (the chat-task contract already honors that marker).
- Inbound photo updates are resolved to local files. The poller picks the largest `PhotoSize`, calls `getFile` + `downloadFile`, writes the bytes to `<instanceRoot>/inbound/<bridgeId>/<update_id>-<file_id>.<ext>` (directory mode `0o700`), and prefixes the task input with a `[photo: <path>]` header so the agent's file toolset can inspect the attachment without changing how task inputs flow through the runtime. The inbound `MessagingMessageRecord.media` records both the local path and the originating `file_id`.
- While an inbound-Telegram task is non-terminal, the poller fires `sendChatAction("typing")` for the originating chat on a ~4-second cadence (Telegram chat actions auto-clear after ~5 seconds). The pulse runs detached so a slow `sendChatAction` does not block the next update batch, and halts on the first error so a revoked chat or network blip cannot keep it looping. The reply mirror still waits for the underlying task to reach terminal state before syncing, so a single typing error does not drop the assistant's response.
- The detached typing+reply worker is tracked in a per-supervisor set; `stopAll` drains the set with a bounded timeout. `sendChatAction` accepts and threads an `AbortSignal` so typing-pulse fetches cancel in-flight when the supervisor aborts; `sendMessage` and `sendPhoto` still don't, so the drain cap is the upper bound on shutdown latency for an in-flight outbound dispatch.
- SIGTERM aborts every active long-poll via `AbortController` so shutdown does not wait out the 25-second timeout.
- The Telegram HTTP client (`src/integrations/telegram.ts`) is mockable via an injected `fetch`; the messaging module exposes `setMessagingDeps` for tests to substitute a stub `TelegramClient`. Production callers leave both unset.

## Security Boundary

- The bot token is a write-only field on the create payload. It is encrypted at rest and never re-emitted — neither on the bridge record, nor in audit evidence, nor on `MessagingMessageRecord`. A bridge owner has no API to read it back; they re-supply it by recreating the bridge.
- The poller calls `receiveMessagingInput` which submits a chat-task. Task-level approval (per ADR `approval-execution-abort.md`) and the active-agent toolset/messaging-target filters (per `agents-replace-profiles.md`) apply unchanged. The bridge does not bypass them. The reply-mirror dispatches the assistant message back via `sendMessagingOutput`, which honors the same messaging-target filter — an agent restricted from a target via its messaging-target filter cannot use the bridge to escape that restriction.
- `allowed_updates` is pinned to `["message"]`. Callback queries, inline events, edits, and other update kinds are not surfaced until the runtime grows explicit routing for them. An attacker controlling a Telegram chat cannot reach internal call sites by sending update kinds the runtime ignores.
- The bridge owner controls which chats can speak to the bot via the chat allowlist on `bridge.metadata.allowedChatIds`. The poller silently drops inbound updates from chats that are not enrolled (and still advances Telegram's update offset so denied updates can't accumulate). There is NO trust-on-first-use: even the operator's first DM is denied until they either run `gini messaging allow <bridge> <chat-id>` or claim the bridge's auto-minted pairing code. Task-level approval (per ADR `approval-execution-abort.md`) and the active-agent toolset / messaging-target filters (per `agents-replace-profiles.md`) apply on top of the allowlist for every inbound message that makes it through.

## Open Questions

- Group privacy mode. Telegram bots default to "privacy mode" in groups, meaning Telegram only delivers messages that mention the bot, reply to the bot, or invoke a command. Users who want a fully-listening bot must disable privacy in @BotFather (`/setprivacy` → Disable); we can't toggle this from the code path. The current implementation works in both modes — privacy-on is the safer default since it cuts noise and inference costs in busy chats. Note that even with privacy off, the chat allowlist still gates which chats the bot will respond to; privacy mode is defense-in-depth on top of the allowlist, not a replacement for it.
- Owner rotation. Today `ownerChatId` is implicit (whoever DM'd first) and there's no flow to transfer ownership to a different chat. A future `gini messaging set-owner <bridge> <chat-id>` would let a team rotate the owner without recreating the bridge. The current `allow` / `deny` pair is sufficient for the single-user case the bridge is designed around.
- Other media kinds. Documents, voice notes, videos, and stickers are not surfaced today. Each follows the same pattern as the photo path (extend `IncomingPayload`, add a `getFile` + `downloadFile` call, save under the inbound directory) but each carries its own caption/duration/mime semantics worth handling in a focused follow-up.
- Inline-keyboard buttons and callback queries. Approval prompts would benefit from inline keyboards; both require expanding `allowed_updates` and a dispatch table from `callback_query.data` to runtime actions.
- Vision-aware task input. Inbound photos currently surface as a `[photo: <path>]` header in the task input string. A multimodal provider could consume the file directly via a content block; wiring that requires changes to `Task.input` (which is `string` today) or to the chat-task content shape.
- Network egress fallbacks. Some networks block the public IP `api.telegram.org` resolves to. A hostname-preserving fallback transport (curl `--resolve` equivalent) is a plausible future addition but is not required by the local-first shape.
- Richer markdown fidelity. The formatter only recognizes `**bold**`, inline code, and fenced code blocks today. `_italic_` markers come through as literal underscores (the underscore is ambiguous with `snake_case` identifiers that the agent emits constantly, so naive conversion would break code-adjacent prose). Markdown headers, list bullets, blockquotes, and `[text](url)` links also pass through as escaped literals; Telegram MarkdownV2 has no native list primitive at all, so even an HTML-mode rewrite wouldn't recover bullets. Bare URLs auto-link on Telegram's side, so `https://example.com` works without any markdown sugar.
- Outbound message length. Telegram caps `sendMessage` payloads at 4096 characters; we don't chunk yet. A long assistant summary fails with `Bad Request: message is too long`, the error lands on the outbound `MessagingMessageRecord`, and the user sees nothing in Telegram. Chunking on paragraph or sentence boundaries (and possibly attaching the long form as a document) is a focused follow-up.

## Verification

- `bun test src/integrations/telegram.test.ts` exercises the HTTP client (`getMe`, `sendMessage`, `sendChatAction`, `sendPhoto` for all three source variants, `getFile`, `downloadFile`, `getUpdates`, and error paths) against an injected `fetch`.
- `bun test src/integrations/telegram-format.test.ts` covers the MarkdownV2 formatter: bold conversion, code-span preservation, exhaustive special-character escaping, and the bold-wrapping-code regression case where ``**`cmd`**`` must render as a code span without leaking literal `**`.
- `bun test src/integrations/messaging.test.ts` exercises add/health/send/disable for telegram-kind bridges against an injected `TelegramClient`, including secret-store round-trip, MarkdownV2 dispatch, photo sends, error propagation, allowlist enforcement (every chat denied on a fresh bridge — no TOFU), recent-denied accumulation with dedupe-by-chatId, the `allowChat` / `denyChat` round-trip (including owner-chat persistence across deny and removal from the recent-denied list on allow), and the pairing-code lifecycle (auto-mint on bridge creation, claim with right code → enroll + consume, wrong code → reject, expired code → cleanup, group chats reject codes, `pairMessagingBridge` rotates).
- `bun test src/integrations/telegram-poller.test.ts` exercises the supervisor's start/stop reconciliation, inbound text routing (incoming text → task submission → offset advance), inbound photo capture (download → save under instance root → `[photo:]` header), the typing-pulse, disable-cleanup, group-chat handling end-to-end (mention stripping, `<sender>: <message>` prefix, and `reply_to_message_id` threading on the mirrored reply), pairing-code claim through the poller (private DM with valid code → enrollment + "Paired." confirmation, no task created), and silent-drop of stranger updates (no `messagingMessageRecord` created, the chat lands on `recentDeniedChats`, and `metadata.lastOffset` still advances).
