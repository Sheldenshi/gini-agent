# Email Watch

## Decision

An email watcher is an `EmailWatcherRecord` (per `(account, sender-query)`) on
`RuntimeState`. Each watcher is driven by a backing interval-driven scheduled job
whose **pre-run hook** (`handlerId: "gmail-delta"`, see
[Pre-LLM Job Hooks](job-pre-run-hooks.md)) runs the delta engine
(`src/integrations/gmail-poll-worker.ts`) before any model turn: it polls the
user's Gmail through the existing `gws` CLI for new matching message ids, dedups
them, applies a deterministic safety floor, and COLLECTS a fenced draft prompt
per surviving new match. The hook maps the result onto the scheduler's typed
contract — **zero matches → short-circuit (no model turn); one or more → the
drafting turn runs in the watcher's dedicated chat session with the fenced
matches injected as context**. The woken agent reads the full message and
composes/sends a reply via the EXISTING `google-gmail` skill — the hook reads
ONLY metadata (From/Subject/Date/snippet) and never message bodies.

The architecture is deliberately a **thin deterministic detection floor + skill-driven action**, now riding the general pre-run job-hook primitive rather than a bespoke loop:

- **Trigger floor (the `gmail-delta` pre-run hook):** poll `gws` → dedup → safety-filter → return matches as context. Zero model turns when there's no new matching mail (the run short-circuits before the model).
- **Action (reuse the skill):** the drafting turn reads the body and drafts/sends via `google-gmail`, gated by `terminal_exec`'s existing approval.

No native Gmail REST client, no new OAuth flow, no new token store, and no new
`gmail_send`/`gmail_draft` tools are introduced. The watcher only writes a
config record; everything side-effecting flows through the already-approval-gated
skill.

## Context

This follows the messaging-bridge ADRs ([telegram-bridge.md](telegram-bridge.md),
[discord-bridge.md](discord-bridge.md)) and the connector-reprobe maintenance
loop. The user asked for an email-watching feature with multi-account support,
sender filtering, and drafted responses, and chose to reuse the already-set-up
`gws` (`google-gmail`) skill rather than build a native Gmail client.

`gws` holds ONE signed-in Google identity per machine, so v1 watches that single
account. The data model is multi-account-SHAPED (`provider` + `accountEmail` +
`credentialName` per record) so a later phase can fan out to per-account
credentials (gws multi-profile or native tokens) without a schema migration.

Reference systems shaped the design: OpenClaw's untrusted-content wrapping (borrowed),
Hermes's layered inbound-safety + startup watermark seeding (borrowed), and the
"separate email-as-connector from watching-as-scheduler" lesson from Claude Code.
OpenClaw's Pub/Sub-push infrastructure and Hermes's auto-send + plaintext creds
were rejected as the wrong shape for a local-first, consent-gated runtime.

## Required Now

- `EmailWatcherRecord` carries `{ id, instance, agentId?, provider:"gmail", accountEmail?, credentialName?, query, labelIds?, lastSeenInternalDate?, chatSessionId?, jobId?, enabled, status:"ok"|"error"|"needs_auth", lastError?, lastPolledAt?, createdAt, updatedAt }`. `jobId` back-references the backing scheduled job that drives the watcher. State CRUD lives in `src/state/email-watchers.ts`; all writes go through `mutateState`, never the JSON file directly.
- Dedup is an `email_seen (instance, watcher_id, message_id, seen_at)` SQLite table in `memory.db` (schema version 10) with `markEmailSeen` / `isEmailSeen`. `markSeen` is per item so a crash mid-batch never replays an already-handled email. Retention: rows live for the life of the watcher; removing a watcher drops its rows via `deleteEmailSeenForWatcher` so dedup state never outlives the record it belongs to.
- Each watcher's backing job is an interval-driven scheduled job (60s) bound to the watcher's dedicated chat session, with a `preRunHook = { handlerId: "gmail-delta", config: { watcherId } }`. The single scheduler (`runDueJobs`, the 1s self-rescheduling loop) is the only driver — there is no dedicated gmail loop and no `GINI_GMAIL_POLL_MS` knob; the cadence is the job's `intervalSeconds`. The in-flight hook + its drafting turn run inside `runDueJobs`, already covered by the existing SIGTERM scheduler drain. Creating a watcher provisions the job (rolled back if job creation fails); removing it removes the job AND the dedicated chat session; disable/enable pause/resume it. A startup `backfillEmailWatcherJobs` provisions a backing job for any enabled watcher missing a resolvable `jobId` (idempotent self-heal). Provisioning is idempotent against the crash-between-create-and-stamp window: before creating, both `addEmailWatcher` and `backfillEmailWatcherJobs` ADOPT an existing job whose `preRunHook.config.watcherId` matches (re-stamping `jobId`) instead of creating a duplicate, so a watcher never ends up double-polling.
- Before polling, the `gmail-delta` hook checks `gws auth status` (via `gwsSessionStatus`). When signed out, the watcher flips to `needs_auth` and the hook short-circuits (no spam, NO model turn); the backing job stays active and the next tick retries once the user re-auths. (Signed-out is never surfaced as a hook `error`, which would fail the job and stop scheduling.)
- The hook spawns `gws` exactly as `gws-session.ts` does for `gws auth status`: `zsh -lc`, `stdin: "ignore"`, kill-on-timeout, inheriting `process.env`. stdout AND stderr are drained CONCURRENTLY (an unread piped stream can fill its ~64KB OS buffer and deadlock the child until the kill timer fires). The subprocess boundary is injectable so unit tests stub it. `gws` prints a `Using keyring backend:` preamble to STDERR before its JSON, so stdout begins at the first `{`; the parser still defensively skips to the first `{`.
- Detection uses `gws gmail users messages list --params '{"userId":"me","q":"<query>","maxResults":N}' --format json --page-all --page-limit P`. Gmail lists newest-first, so for a steady-state window that fits within the page cap the worker enumerates the WHOLE matching window across pages (NDJSON, one object per page) and drains it oldest-first rather than the newest page only — otherwise older matches that don't fit on the first page would fall below the advancing watermark. When the page cap IS hit the window isn't fully enumerated, and the seeding/truncated-window handling below baselines past the un-listed tail rather than draining it. Once a watcher has a watermark the query is bounded with `after:<epochSec>` derived from `lastSeenInternalDate`, so steady-state polling lists almost nothing.
- Metadata for the safety floor + prompt comes from `gws gmail users messages get --params '{"userId":"me","id":"<id>","format":"metadata","metadataHeaders":["From","Subject","Date"]}'`. Bodies are NEVER fetched here.
- Safety floor (deterministic, drop-at-trigger): drop automated senders (From matches `no-reply` / `noreply` / `mailer-daemon` / `postmaster` / `bounce` / `notifications@`) and drop self (the address parsed out of the From header equals the signed-in address from `gws gmail users getProfile`, compared by EQUALITY — not substring, which would false-drop a human whose address merely contains self's). Dropped messages are still `markSeen` so they're never reconsidered.
- On a surviving match the hook COLLECTS a fenced draft prompt (it no longer spawns a turn per match). The hook returns the collected prompts as `context` items; the scheduler then runs ONE drafting turn for the watcher's job (`dispatchPromptRun`, the same path scheduled jobs use, `mode: "chat"` bound to the watcher's session) with all surviving matches for that tick injected as context — bounded by `MAX_MESSAGES_PER_TICK`, so one tick injects at most that many fences into one turn rather than spawning N concurrent turns. Each prompt instructs the agent to `read_skill google-gmail`, read the full message, post a PROPOSED reply in this chat for review, and NOT send unless the user says so; it must emit exactly `[SILENT]` when nothing is actionable (the existing chat-task suppression sentinel). When the hook collects zero prompts the run short-circuits with no model turn at all — strictly better than the prior post-hoc `[SILENT]` suppression.
- The matched email metadata in the prompt is wrapped in an `UNTRUSTED_EMAIL_METADATA` fence (the prompt-injection boundary): everything inside is quoted data the agent must not treat as instructions; the trusted instructions live outside the fence. The PRIMARY breakout defense is the JSON encoding — the untrusted fields are emitted as a single JSON object inside the fence, so quotes/newlines/markers are escaped and the whole payload stays on one physical line and cannot break the container even if a sentinel-like substring survives. Belt-and-suspenders: each field is also stripped of fence-sentinel substrings (the strip loops to a fixpoint, so a nested rejoin like `END_UNT…RUSTED…` cannot re-form a sentinel) and has CR/LF collapsed, and the fence delimiter carries a per-message nonce (derived deterministically from the message id) so the close token can't be guessed and forged from inside the data.
- Startup seeding BASELINES, it does not enumerate. The `after:` watermark only ever moves FORWARD (toward newer mail), so an older un-listed tail can never be reached by advancing it — a seeding path that depends on enumerating the whole backlog would be wrong on a large inbox (gws lists only the newest page-cap worth). Instead, on a watcher with no `lastSeenInternalDate` the first fire takes the NEWEST listed id, fetches only its `internalDate`, sets the cursor there, `markSeen`s just that one boundary id (plus any sibling sharing its exact epoch second, since `after:` is inclusive of the boundary second), and collects NO prompt — so the run short-circuits with no model turn (or, if the inbox had no matches, baselines the cursor at `now`). Pre-existing mail older than the baseline is excluded by `after:` forever — which is the correct behavior (never draft a backlog), regardless of inbox size, with no pagination dependence on first run.
- Steady-state draining of a fully-enumerated (non-truncated) window is OLDEST-FIRST. Each fire enumerates the window, drains it oldest-first, caps the matches collected at `MAX_MESSAGES_PER_TICK = 25`, and advances the cursor ONCE at the end to the LAST CONSUMED item's `internalDate` — forward progress that drains a >cap backlog over successive ticks without ever stepping past an un-consumed match. Crash safety is the `email_seen` store, not the cursor: dropped items are `markSeen` inline, and a surviving (to-be-drafted) match's `markSeen` + the cursor advance are DEFERRED to a commit thunk that runs only after the drafting turn dispatches (see the delivery note below), so a crash before delivery re-lists the window and dedup skips whatever was already handled. When the cap is hit with items remaining, the hook logs a warning and the remainder drains on later ticks.
- A TRUNCATED steady-state window (more genuinely-new matches than the page cap of `WINDOW_PAGE_LIMIT * WINDOW_PAGE_SIZE` since the cursor — only reachable after downtime + high volume) is a **bounded skip-with-notice, not an eventual drain**. Because the older tail isn't listed and the forward-only watermark can never reach it, the hook does NOT silently advance past the tail and does NOT storm a draft per message: it jumps the cursor to the NEWEST listed match, `markSeen`s that boundary id, logs the truncation, and collects exactly ONE notice prompt (gated on the newest id not already being seen, so it fires at most once per backlog episode) telling the user a large backlog accumulated and that it isn't drafting replies to all of them (it can triage on request). Zero overflow drafts, no infinite re-loop.
- Delivery is **exactly once on the happy path, at-least-once across the delivery boundary**: a surviving match's `markSeen` + cursor-advance are not committed when the prompt is collected — they are deferred into a commit thunk the scheduler awaits ONLY after `dispatchPromptRun` has spawned the drafting turn. A throw before delivery (a mid-poll list/get failure, or a `dispatchPromptRun` spawn failure after collection) leaves those ids un-seen and un-cursored, so they re-trigger on a later healthy fire. Committing before dispatch (the older markSeen-after-collect-but-before-spawn order) would silently lose a match whose drafting turn failed to spawn. Intentional skips with no delivery (dropped/safety-floored items, the seeding baseline, the truncated-window notice) still commit inline, since there is nothing to deliver.
- A gws/transport failure marks THAT watcher `status: "error"` with a scrubbed `lastError` (absolute `.json`/`.enc` paths and any home-rooted path redacted) and the hook SHORT-CIRCUITS — it never returns the hook `error` kind, because that would flip `job.status="failed"` and stop the backing job from scheduling. The watcher keeps its active job and retries next tick; the next signed-in / healthy tick clears it back to `ok`. The hook `error` kind is reserved for broken config (a missing/unknown watcher pointed at by the job).
- The `email_watch` agent tool (toolset `email`, action enum `add | list | remove | disable | enable`) is always-on and low-risk — it only writes a watcher record (and, on `add`, a dedicated chat session). `disable` / `enable` pause/resume a watcher (and its backing job) without deleting it. One tool with an action enum keeps the catalog surface minimal, mirroring the `create_job` / `list_jobs` / `delete_job` precedent collapsed into one.
- `add` builds the query: a raw `query` wins; else `from:<sender> is:unread`; else `is:unread`. The watcher + its dedicated chat session are created in ONE `mutateState` write (no orphan session on failure), then `addEmailWatcher` provisions the backing job and stamps `EmailWatcherRecord.jobId`; if job creation fails the watcher is rolled back so no orphan watcher/session lingers. `remove` deletes the backing job first (so the scheduler stops firing it), then the watcher row + its dedicated chat session (an auto-created job channel that must not outlive the watcher — `deleteChatSession` cascades its messages/blocks/identity snapshot, guarded if the session is already gone), then its `email_seen` rows. The tool, `POST /api/email/watchers`, and `gini email add` all route through the shared `addEmailWatcher` helper so they produce identical records.
- `disable` / `enable` route through `setEmailWatcherEnabled`, which flips `EmailWatcherRecord.enabled` AND pauses/resumes the backing job (`updateJobStatus`) so the scheduler stops/starts claiming it; the `gmail-delta` hook also self-guards on `!enabled` (defense in depth against a paused-but-claimed race).
- HTTP: `GET /api/email/watchers` (list), `POST /api/email/watchers` (add), `PATCH /api/email/watchers/:id` (toggle `{ enabled }`), `DELETE /api/email/watchers/:id` (remove) — thin handlers over the state module. CLI: `gini email list | add --from <sender> [--query <q>] | remove <id> | disable <id> | enable <id>`.

## Trust Boundary

- Inbound email is untrusted external content. The `gmail-delta` hook fences matched metadata in each collected prompt so a hostile subject/snippet cannot smuggle instructions to the agent; the fence is opaque to the scheduler (the hook returns `untrusted:false` because it owns the JSON+nonce fence — see [Pre-LLM Job Hooks](job-pre-run-hooks.md)). Bodies are read only inside the drafting turn, through the consent-gated `google-gmail` skill — the deterministic hook never touches them. The hook runs outside the spawned task's approval/audit/trace envelope, so it is read-only by contract (it writes only the watcher cursor + `email_seen`).
- Drafting a reply is surfaced as a CHAT MESSAGE in the watcher's session for the user to review; the agent does not auto-send. Sending requires an explicit user instruction, which then runs `gws gmail +reply` through `terminal_exec`'s approval gate — no new send tool or risk plumbing is added.
- The hook shells `gws` with fixed query strings and integer `after:` values it builds itself; raw email bytes never reach the shell. `gws` reads its own client config + token from `~/.config/gws/`, so no credentials are injected into the spawn env and none land on the watcher record.

## Limitations / Open Questions

- **Single account (v1).** `gws` holds one signed-in identity, so v1 watches that account. The data model is multi-account-shaped; true multi-account needs gws multi-profile or native per-account tokens (a future phase).
- **`after:`-second cursor precision.** Gmail's `after:` is second-granular while the watermark is millisecond `internalDate`. The `email_seen` dedup makes the second-rounding harmless (a re-listed boundary message is skipped), but it does mean the bounded query can re-surface same-second messages, which dedup then drops.
- **History robustness.** v1 uses `messages.list` + a per-watcher watermark rather than `history.list`, so there's no `historyId`-404 full-resync path to maintain. If a future phase moves to `history.list` for lower latency/cost, it must add the bounded full-resync on a 404 history id.
- **Drafts in Gmail.** v1 posts the proposed reply as a chat message, not a Gmail draft. Landing drafts in the user's real Gmail Drafts (threaded) is a future option.
- **Pub/Sub-pull low-latency.** Polling at ~60s is the v1 cadence. A future opt-in could use Pub/Sub-pull (outbound only, no public webhook) for lower latency without changing the trust model.

## Verification

- `bun test src/integrations/gmail-poll-worker.test.ts` exercises the pure delta-engine helpers: the message-list JSON parse (incl. the `gws` preamble and multi-page NDJSON window), metadata parse, the safety floor (automated + self dropped, with no false-drop of a human whose address contains self's), the untrusted-metadata fence under a breakout attack and a nested-rejoin attack (hostile subject/snippet carrying the close sentinel + injected instruction stay escaped inside the JSON data container; exactly one physical line is the legitimate nonce-suffixed close marker), and the error path scrub.
- `bun test src/jobs/hooks/gmail-delta.test.ts` exercises the watcher-state regimes through the `gmail-delta` hook entrypoint with an injected `gwsSpawn` + ephemeral instance dir + ephemeral `memory.db`: startup seeding (short-circuit, no turn on first fire), dedup + cursor advance (surviving a simulated restart), the oldest-first backlog drain + cap (cursor advances to the last consumed item, the remainder drains on later fires), the truncated-window single notice (cursor jumps to newest, no re-loop, at-most-once even with a bad newest metadata-get), same-second seeding siblings, the `after:<epochSec>` watermark bound (present once a cursor exists, absent on seeding), at-least-once delivery on a poll failure (re-triggers on the next healthy fire), the signed-out → `needs_auth` path (job stays active), a gws error → watcher `error` + scrubbed message + short-circuit (job NOT failed), and config-broken → `error`.
- `bun test src/jobs/hooks/hooks.test.ts` covers the general pre-run hook primitive (see [Pre-LLM Job Hooks](job-pre-run-hooks.md)). `bun test src/state` covers the `EmailWatcherRecord` CRUD + `email_seen` helpers.
